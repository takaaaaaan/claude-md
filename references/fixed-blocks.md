# A ブロック逐語テキスト

`claude-md-init` / `claude-md-align` が CLAUDE.md に **そのまま貼り込む** 固定セクション。`<...>` のプレースホルダだけ、検出値で置き換える。順番・位置は [`format.md`](./format.md) の骨格に従う。

---

## セクション 0 — 冒頭の一文

```markdown
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.
```

## セクション 1 — メンテナンス方針

```markdown
## 重要: このファイルのメンテナンス方針

- このリポジトリで開発が進み、**全体構造やフローが変わった**、または **CLAUDE.md に書かれている内容と実態がズレた**ことに気づいたら、CLAUDE.md を更新する。
- ただし **CLAUDE.md の変更は必ず user の許可を取ってから行うこと**。Claude 側の判断で勝手に書き換えない。「○○の節を××のように更新したい(理由: …)」と提案 → user の "OK" を待ってから編集する。
```

## セクション 2 — 協働とコミュニケーション（報連相）

```markdown
## 協働とコミュニケーション（報連相）

Claude（AI）もこの開発チームの一員。**報告・連絡・相談（報連相）** を大事にし、勝手に突き進まず user と歩調を合わせる。

- **ドキュメントと実態のズレは報告する。** この CLAUDE.md や他のドキュメントに書かれた内容と実際のコードが食い違っていたら、黙って直さず「ここがズレている(該当箇所)」と user に報告する。
- **ドキュメントを鵜呑みにしない。** 記述が常に正しいという前提を置かない。動作・挙動が説明と違う疑いがあれば、実際に確認したうえで user に報告する。
- **user がすべて把握している前提を置かない。** 分かりにくそうな点・伝わっていなさそうな点は、こちらから能動的に説明・確認・報告する。
- **手動の方が良い作業は user に任せる。** 認証ログイン・本番反映・破壊的操作・外部公開など、user 自身がやった方がよい部分は Claude が代行せず、手順を示して user に実行を提案する。
```

## セクション 5 — 作業ルール: worktree とポート（必須）

`<DEFAULT_PORTS>` と `<PORT_RANGE>` は解析で検出した実値に置換する（例: FE なら `3000` / `3100,3200…`、FastAPI BE なら `20000`（docker host マッピング）/ `20000–21000`）。複数サービスがあれば各々について書く。

```markdown
## 作業ルール: worktree とポート（必須）

修正・新機能・実験など **コードに手を入れる作業は、必ず git worktree を切ってから** 進める。メインの作業ツリーで直接いじらない。完了後に ff-merge で戻す。

各 worktree でアプリ/サーバを起動してテストするときは、**他の worktree やデフォルト設定と衝突しないポートを一時的に使う**：

- **デフォルトポート（<DEFAULT_PORTS>）は使わない。** 予約レンジ <PORT_RANGE> 内の別ポート（例: …）を一時的に使う。
- worktree ごとに番号帯を分けて、複数 worktree を同時に立てても衝突しないようにする。

起動したら **user が確認しやすいように URL を提案して伝える**（例: 「http://localhost:XXXX で起動しました。確認してください」）。

**テストが終わったら、起動したプロセス（dev サーバ / uvicorn / docker compose / streamlit など）は必ず停止する。** 立てっぱなしにしてポートを占有しない（`docker compose down` 等）。
```

## セクション 10 — Commit style

```markdown
## Commit style（strictly enforced）

`<emoji> <type>: <description>` 形式。**コミットメッセージは韓国語で書く**(インフラ・config 系のみ英語可)。

Types: `✨ feat` `🐛 fix` `♻️ refactor` `🧪 test` `📝 docs` `🔧 chore` `🚀 perf` `💄 style` `🔒 security` `🗑️ remove` `🚧 wip`。

`git add -A` / `git add .` は禁止 — パスを明示してステージする。サブモジュールがある場合は先にサブモジュール内でコミットし、親リポではポインタ更新を別コミットにする。
```

## セクション 11 — ファイル変更時のコメント規約

```markdown
## ファイル変更時のコメント規約

昔ながらの流儀で、ファイルには「役割」と「変更の意図」をコメントで残す。

- **ファイルの役割**: 新規作成時や責務が変わったときは、ファイル先頭付近に「このファイルが何のためのものか」をコメントで書く。
- **変更の意図**: 重要な変更には「いつ・誰が・なぜ・何を」変えたかを、意図とともにコメントで残す（例: `// 2026-06-12 ueno: KMA 欠測時のフォールバックを追加(理由: Seoul で miss するため)`）。
- コミットログと重複しても、コードを読む人がその場で経緯を追えるよう、コメント側にも残す。
```

## CLAUDE.local.md テンプレ（任意・個人ローカル）

`CLAUDE.md` を確定したあと、**user が希望した場合だけ**生成する。`CLAUDE.md` のセクションではなく、リポジトリ直下に置く **別ファイル**。**コミットしない**個人ローカル専用の控え。生成と同時に `.gitignore` へ `CLAUDE.local.md` を登録する（未登録なら追記を提案、既にあれば触らない）。

ローカルのポート控えは、検出できた実値や user 申告のポートで穴埋めしてよい（無ければ例のまま残す）。個人メモは user に書いてもらう前提で、見出しと書き出しの例だけ置く。

````markdown
# CLAUDE.local.md

このファイルは **個人・マシンローカル専用** の控え。`CLAUDE.md`（チーム共有・コミット対象）とは別に、**コミットしない**個人的な設定・メモを置く。`.gitignore` 登録済みで、リモートや他メンバーには共有されない。

## 個人・マシン依存メモ

- このマシン固有のパス・環境（例: ローカルの絶対パス、個人の venv / ツールの場所、エディタ設定など）
- 個人的に使っているローカル専用コマンド・エイリアス・ショートカット
- 自分用の作業メモ・好み（チームには強制しないもの）

## ローカルのポート / URL 控え

- 自分が今立てている worktree / サービスのポート・ローカル URL の控え（例: `feat-x worktree → http://localhost:3105`）
- 衝突を避けるための自分用の番号帯メモ
````

`.gitignore` には次の 1 行を入れる（既にあれば追記しない）：

```gitignore
CLAUDE.local.md
```
